   178→### Design Decisions
   179→
   180→- **Pagination format**: `{ data, total, page, pageSize, totalPages }` for all paginated endpoints
   181→- **Non-paginated endpoints** return `{ data }` directly
   182→- **Search**: Uses Prisma `contains` filter (SQLite is case-insensitive by default)
   183→- **Error handling**: try/catch with console.error and appropriate HTTP status codes (400, 404, 500)
   184→- **Validation**: Contact POST uses Zod `contactMessageSchema` from `@/lib/validations`
   185→- **Constants**: Uses `DEFAULT_PAGE_SIZE` from `@/lib/constants`
   186→- All imports use `import { db } from '@/lib/db'`
   187→- Responses use `NextResponse.json()`
   188→- No admin POST/PUT/DELETE routes created (will be done separately)

   820→### API Route Implementation Details
   821→
   822→All API routes follow a consistent pattern:
   823→- **GET (list)**: Paginated with `page`, `pageSize` query params, returns `{ data, total, page, pageSize, totalPages }`
   824→- **GET (single)**: Returns `{ data: item }` or 404
   825→- **POST**: Validates body with Zod schema from `@/lib/validations`, auto-generates slug via `generateSlug()` from `@/lib/helpers`, returns 201 on success with `{ success: true, data }`
   826→- **PUT**: Same validation as POST, checks existence first (404 if not found), handles slug collision with `slug-timestamp` pattern
   827→- **DELETE**: Checks existence first (404 if not found), returns `{ success: true, message }`
